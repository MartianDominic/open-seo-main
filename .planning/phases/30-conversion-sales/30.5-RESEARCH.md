# Phase 30.5: Prospect Pipeline Automation - Research

**Researched:** 2026-04-22
**Domain:** Prospect pipeline automation, CSV import, bulk operations, scheduling
**Confidence:** HIGH

## Summary

Phase 30.5 enables agencies managing 500+ prospects to automate their sales pipeline with CSV bulk import, batch analysis queueing, automatic priority scoring, pipeline stage automation, and bulk UI actions. The existing codebase provides a strong foundation: prospects schema with `priorityScore` field already exists (Phase 26), BullMQ queue infrastructure is in place (`prospect-analysis` queue), priority scoring algorithm is implemented (`calculatePriorityScore.ts`), and automation patterns exist from Phase 30-08 proposals.

The research validates that all required libraries are already installed: papaparse (5.5.3) for CSV parsing, @tanstack/react-table (8.21.3) for bulk selection UI, recharts (3.8.1) for pipeline visualization, zod (4.3.6) for validation, and resend (6.12.0) for digest emails. The existing `AnalysisService.bulkQueueAnalysis()` method already handles quota management with 10 analyses/day/workspace limit.

**Primary recommendation:** Extend existing infrastructure rather than building new - add CSV import endpoint using papaparse, expand `bulkQueueAnalysis` for 100+ prospects, add pipeline_stage column to prospects table, create automation rules schema similar to proposals automation, and build bulk actions UI using TanStack Table row selection.

### Pattern 1: CSV Import with Preview
**What:** Parse CSV client-side for preview, then POST raw rows to server for validation and import
**When to use:** User uploads CSV file, wants to preview before committing
**Example:**
```typescript

function handleFileSelect(file: File) {
  Papa.parse(file, {
    header: true,
    skipEmptyLines: true,
    complete: (results) => {
      // Preview parsed rows
      setPreviewRows(results.data.slice(0, 10));
      setTotalRows(results.data.length);
      setRawCsvData(results.data);
    },
    error: (error) => setParseError(error.message),
  });
}
```

### Anti-Patterns to Avoid
- **Processing CSV server-side on upload:** Parse client-side for preview, POST already-parsed JSON to server. Don't stream large files through server memory.
- **Ignoring quota limits:** Always check `getRemainingAnalysesToday()` before queueing bulk analysis.
- **Mutating row selection state directly:** Use TanStack Table's `onRowSelectionChange` callback.
- **Hardcoding automation rules:** Store in database, not code - allows workspace customization later.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CSV parsing | Custom delimiter/quote parser | papaparse | RFC 4180 edge cases, encoding, BOM handling |
| Row selection state | Manual checkbox tracking | TanStack Table rowSelectionFeature | Handles all/some/none states, pagination |
| Bar charts | Custom SVG | Recharts BarChart | Responsive, tooltips, animations |
| Job scheduling | setInterval/setTimeout | BullMQ repeatable jobs | Persistent across restarts, Redis-backed |

## Common Pitfalls

### Pitfall 1: CSV Encoding Issues
**What goes wrong:** Non-UTF8 files (Latin-1, Windows-1252) produce garbled characters
**Why it happens:** Users export from Excel with regional settings
**How to avoid:** Use papaparse `encoding` option, auto-detect with BOM
**Warning signs:** Names with diacritics showing as ???

### Pitfall 2: Quota Exhaustion Mid-Batch
**What goes wrong:** User selects 200 prospects, only 10 get queued, rest silently skipped
**Why it happens:** Daily quota (10/workspace/day) is enforced per-request
**How to avoid:** Show remaining quota before action, return `{queued, skipped}` response, inform user
**Warning signs:** User complaints about "analysis not running"

### Pitfall 3: Duplicate Domain Import
**What goes wrong:** CSV contains duplicate domains, some fail silently
**Why it happens:** Unique constraint on (workspace_id, domain)
**How to avoid:** De-duplicate in CSV parser before insert, use upsert or skip-on-conflict
**Warning signs:** Import reports fewer created than CSV rows

### Pitfall 4: TanStack Table Indeterminate Checkbox
**What goes wrong:** "Select all" checkbox doesn't show indeterminate state
**Why it happens:** Radix Checkbox uses `indeterminate` prop, not `checked="indeterminate"`
**How to avoid:** Use ref callback: `ref={(el) => el && (el.indeterminate = table.getIsSomeRowsSelected())}`
**Warning signs:** Header checkbox only shows checked/unchecked, not partial

### Pitfall 5: Recharts ResponsiveContainer Height
**What goes wrong:** Chart collapses to 0 height
**Why it happens:** ResponsiveContainer needs explicit height or parent with defined height
**How to avoid:** Set explicit `height={300}` or ensure parent has CSS height
**Warning signs:** Empty space where chart should be

### CSV Import Server Function
```typescript
// Source: Existing serverFunctions/prospects.ts pattern + papaparse docs
import { z } from "zod";
import Papa from "papaparse";

const csvRowSchema = z.object({
  domain: z.string().min(1),
  companyName: z.string().optional(),
  contactEmail: z.string().email().optional().or(z.literal("")),
  contactName: z.string().optional(),
  industry: z.string().optional(),
  notes: z.string().optional(),
  source: z.string().optional(),
});

export const importProspectsFromCsv = createServerFn({ method: "POST" })
  .middleware(requireAuthenticatedContext)
  .inputValidator((data: unknown) => 
    z.object({ rows: z.array(csvRowSchema) }).parse(data)
  )
  .handler(async ({ data, context }) => {
    const { rows } = data;
    const results = { created: 0, skipped: 0, errors: [] as string[] };
    
    for (const row of rows) {
      try {
        await ProspectService.create({
          workspaceId: context.organizationId,
          ...row,
        });
        results.created++;
      } catch (error) {
        if (error.code === "CONFLICT") {
          results.skipped++;
        } else {
          results.errors.push(`${row.domain}: ${error.message}`);
        }
      }
    }
    
    return results;
  });
```

### Daily Digest Queue
```typescript
// Source: Existing scheduleQueue.ts pattern
export const DIGEST_QUEUE_NAME = "daily-digest" as const;

export const digestQueue = new Queue<DigestJobData>(DIGEST_QUEUE_NAME, {
  connection: getSharedBullMQConnection("queue:digest"),
  defaultJobOptions: { attempts: 3, backoff: { type: "exponential", delay: 5_000 } },
});

export async function initDigestQueue(): Promise<void> {
  const repeatableJobs = await digestQueue.getRepeatableJobs();
  for (const job of repeatableJobs) {
    await digestQueue.removeRepeatableByKey(job.key);
  }
  
  // Run daily at 08:00 UTC
  await digestQueue.add(
    "send-digest",
    { triggeredAt: new Date().toISOString() },
    { repeat: { pattern: "0 8 * * *" }, jobId: "daily-digest" }
  );
}
```

   - Recommendation: Confirm stage names with user before adding to schema

2. **Digest email recipients**
   - What we know: Workspace owners should receive digest
   - What's unclear: Should all workspace members receive it? Configurable per-user?
   - Recommendation: Start with workspace owner only, add preferences later
